## Q1: What are the specific steps and automation options for deploying Run:ai in air-gapped Kubernetes environments?

**Run:ai Self-Hosted Air-Gapped Installation Process** [15][17]:

1. **Obtain Air-Gapped Package**: Request `runai-air-gapped-<VERSION>.tar.gz` from Run:ai customer support
2. **Prerequisites**: 
   - Kubernetes cluster with kubectl access
   - Docker installed on a node
   - Private Docker registry within organization
   - At least 20GB free disk space
3. **Extract and Upload Images**:
   ```bash
   tar xvf runai-airgapped-package-<VERSION>.tar.gz
   kubectl create namespace runai-backend
   export REGISTRY_URL=<Docker Registry address>
   sudo -E ./setup.sh
   ```
4. **Configure Local Certificate Authority**: Required for air-gapped environments; prepare public key of local CA [15]
5. **Install Control Plane**: Use Helm with `custom-env.yaml` generated by setup script [17]
   ```bash
   helm upgrade -i runai-backend control-plane-<VERSION>.tgz \
     --set global.domain=<DOMAIN> \
     --set global.customCA.enabled=true \
     -n runai-backend -f custom-env.yaml
   ```

**Run:ai Professional Services Installer** [16]:
- Comprehensive automation tool with `--air-gapped` flag
- Features:
  - Complete offline installation capabilities
  - Certificate management (self-signed or custom)
  - DNS configuration and CoreDNS patching
  - Nginx Ingress Controller installation
  - Optional Prometheus, GPU Operator, Knative installation
  - Subdomain support with wildcard ingress
- Installation process validates environment, installs prerequisites, detects GPU nodes, labels CPU nodes, deploys backend services

**Control Plane Components** [17]:
- PostgreSQL (data store)
- Thanos (metrics store)
- Keycloakx (identity & access management)
- Grafana (analytics dashboard)
- Redis (caching, disabled by default)

## Q2: How can organizations automate the transfer of container images and Helm charts into air-gapped Run:ai environments?

**Air-Gapped Architecture Pattern** [18]:
```
Internet Zone → Mirror Host → Portable Media → Air-Gapped Zone → Private Registry → Kubernetes Cluster
```

**Tools for Image Mirroring**:

1. **Hauler** [19]: CMU SEI recommends this tool for drastically simplified air-gapped container registry mirroring. Enables transparent image pulling where developers don't need different manifests for air-gapped vs connected environments.

2. **Skopeo** [18]: Command-line tool for image operations
   - Basic mirroring: `skopeo copy docker://source docker://target`
   - Save to tar: `skopeo copy docker://image docker-archive:file.tar`
   - Load from tar: `skopeo copy docker-archive:file.tar docker://registry/image`

3. **Harbor** [18]: Enterprise container registry with:
   - Security scanning
   - Replication policies
   - RBAC
   - Audit logging

**Helm Chart Mirroring** [18]:
1. Download chart: `helm pull <chart> --untar`
2. Extract images: Template chart and grep for image references
3. Mirror each image to private registry
4. Store Helm charts in local Git repository or ChartMuseum

**Best Practices** [19]:
- Configure registry mirror option in Kubernetes for transparent image sourcing
- Developers don't need to maintain different Helm values.yaml files for air-gapped environments
- Standardization speeds deployment time

## Summary

Run:ai provides comprehensive air-gapped deployment support through official tar packages and the Professional Services installer tool. The installation process requires careful preparation of a private Docker registry, local certificate authority, and proper network configuration. 

Automation of image transfer has been significantly simplified by tools like Hauler and Skopeo, which enable transparent registry mirroring. The key architectural pattern involves a mirror host in the internet zone that packages images for transfer via portable media to the air-gapped private registry. Organizations should leverage Helm chart extraction scripts to identify all required images and automate the mirroring process.

The Run:ai installer's `--air-gapped` flag encapsulates much of this complexity, but organizations still need to establish robust transfer processes for ongoing updates and new model deployments.
